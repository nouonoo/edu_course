**Версия:** 1.2  
**Дата:** июнь 2026  
**Окружение тестирования:** Windows 10/11, Chrome/Edge, Frontend :8000, Backend :5000, SQL Server

---

## 1. Шаблон тестового случая

Каждый тест-кейс оформлен по единому шаблону.

### Таблица 14 — Шаблон тестового случая

| Id теста | Название теста | Краткое описание теста |
|----------|----------------|------------------------|
| *значение* | *значение* | *значение* |

| Этапы теста: | Ожидаемый результат: | Фактический результат: | Статус (успешно/завершилось ошибкой) |
|--------------|----------------------|------------------------|--------------------------------------|
| *значение* | *значение* | *значение* | *значение* |

### Таблица 15 — Расшифровка шаблона

| Поле | Описание |
|------|----------|
| **Id теста** | Уникальный ID для каждого тестового примера |
| **Название теста** | Название тестового случая |
| **Краткое описание теста** | Описание того, что должен достичь тест |
| **Этапы теста:** | Перечисление выполненных этапов для достижения цели пошагово |
| **Ожидаемый результат:** | Описание результата, который должен появиться в конце выполнения необходимых этапов |
| **Фактический результат:** | Описание результата после выполнения необходимых этапов |
| **Статус (успешно/завершилось ошибкой)** | Вывод о том, достигнут ли ожидаемый результат |

### Дополнительные обозначения (в описании тестов)

| Обозначение | Значение |
|-------------|----------|
| **P1** | Критический приоритет |
| **P2** | Высокий приоритет |
| **P3** | Средний приоритет |
| **F** | Функциональный тест |
| **N** | Негативный тест |
| **UI** | Тест интерфейса |
| **I** | Интеграционный тест |
| **S** | Тест безопасности |

## 12. Протокол проведения тестирования

### 12.1. Порядок выполнения

1. Подготовка окружения (БД, backend, frontend).
2. Smoke-тест: TC-AUTH-001, TC-COURSE-001.
3. Функциональные тесты по модулям (P1 → P2 → P3).
4. Автоматизированные юнит-тесты.
5. Регрессия после исправлений.

### 12.2. Критерии приёмки

- Все тесты **P1** пройдены на 100%.
- Тесты **P2** — не менее 95%.
- Критические дефекты (блокирующие обучение) — 0.

### 12.3. Запись результатов

Результаты фиксируются непосредственно в каждом тест-кейсе:
- в поле **«Фактический результат»** — что произошло после выполнения этапов;
- в поле **«Статус (успешно/завершилось ошибкой)»** — итоговый вывод.

При обнаружении дефекта дополнительно оформляется баг-репорт с указанием Id теста, даты, тестировщика и шагов воспроизведения.
